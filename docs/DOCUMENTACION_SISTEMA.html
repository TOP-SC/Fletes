Visión y alcance del producto

Un solo sistema que unifica lo que hoy vive en Excel, mails y conocimiento tribal del equipo logístico.

Problema que resuelve

SommierCenter opera con Tango ERP como fuente de pedidos, pero el control de costos logísticos requiere cruzar múltiples fuentes: tarifarios Excel de CLICPAQ, FRANSOF, ALFARO y LBO; prefacturas diarias; reglas de postventa; liquidaciones quincenales; y, para AMBA/GBA, tarifas por km desde sucursales propias. Sin automatización, cada operador reconstruye la lógica en planillas WAMARO — lenta, propensa a error y no auditable.

Objetivo del sistema

  • Centralizar todos los envíos importados desde Tango en una base SQLite con historial por lote.
  • Aplicar reglas de negocio acordadas con Mantello (transporte 40/51/82, zonas, exclusiones).
  • Asignar proveedor por tarifario — no por intuición — con elección manual solo en empates.
  • Conciliar prefactura Clickpack contra costo tarifario calculado (+ seguro fijo $30).
  • Visualizar semáforo de colores para priorización operativa instantánea.
  • Calcular fletes locales desde sucursal más lógica hasta domicilio, con fallback por localidad.
  • Exportar planilla maestro Excel compatible con flujo cliente existente.

Escala, liderazgo técnico y capacidad de equipo

Arquitectura pensada para escalar organizacionalmente

Este sistema no es el output de un tutorial ni de un script aislado. Es el resultado de modelado de dominio logístico real, decisiones de arquitectura desacoplada (API + UI + servicios), convenciones portadas desde Mantello y una UI operativa con reglas de negocio que normalmente requerirían un equipo multidisciplinario: backend, frontend, analista funcional, QA y DevOps. La estructura actual — 25+ servicios, 8 routers, modelos normalizados, parsers tolerantes a Excel sucio, cache geoespacial, migraciones ad-hoc — es la base sobre la que un lead engineer / architect senior podría coordinar un equipo de 100 personas sin reescribir desde cero: cada módulo tiene frontera clara, contrato API explícito y responsabilidad única.

Geocoding, matching y asignación de sucursal

Servicio fletes_matching_service.py — cascada de inteligencia para maximizar casos resueltos (verdes) sin rendirse en la primera lectura de dirección Tango.

Estrategias en orden de ejecución

  1. Variantes de geocoding: dirección completa → domicilio+localidad → calle sin número → solo localidad → localidad+CP
  2. Depósito origen: 14→CD, 12→TH
  3. Alias localidad→sucursal: Pilar→PI, Nordelta→ND, Lanús→LA, Quilmes→QU, etc. + tokens del catálogo
  4. Cercanía geográfica: Haversine sobre 29 sucursales, priorizando alias si coincide
  5. Km estimado: si falla geocode pero matchea localidad — 12 km (misma zona) o 22 km (partido cercano) → zona tarifaria → fila verde con nota

Proveedores de ruta

Nominatim (gratuito, throttle 1.1 s/request) con cache SQLite. Opcional OpenRouteService vía ORS_API_KEY en .env para rutas driving reales.

Limpieza direcciones Tango

Quita prefijos CASA:, DEPTO:, Piso:, normaliza acentos y espacios antes de geocodificar.
